# 단계별 상세 — 단계 × 차원

> [`DESIGN_FLOW.md`](DESIGN_FLOW.md)가 원리(두 축). 이 문서는 각 단계에서 **6개 요구사항 차원**이 어떤 질문이 되고, 그 답이 환경마다 어떻게 갈리는지 푼다.
> 6 차원 = 기능 · 성능·확장 · 신뢰·복구 · 데이터품질 · 보안·거버넌스 · 비용·운영. 각 질문의 *목적*이 곧 그 차원.

---

## 생성 (Generation) — 소스 시스템

> 데이터가 만들어지는 곳. 우리가 통제 못 하는 외부인 경우가 많다 → *조사·계약*의 단계.

| 차원 | 이 단계의 질문 | 답이 갈리는 예 |
|---|---|---|
| 기능 | 무슨 데이터? 우리가 실제 쓰는 필드는? | Criteo 22컬럼 중 timestamp/uid/campaign/click/conversion/cost |
| 성능·확장 | 생성 빈도·속도? 유한(bounded) vs 무한? | 로그=무한 스트림 / 파일덤프=유한 배치 |
| 신뢰·복구 | 소스가 유실·중복을 보장? 재수신(백필) 가능? | DB CDC=순서보장 / 파일=재읽기 가능 |
| 데이터품질 | 스키마 있나? drift 통지? 결측 규약? | Criteo: 전환없음=`-1` 센티넬, 스키마 무통보(외부) |
| 보안·거버넌스 | PII 포함? 소유자·SLA? | uid 개인식별 여부 → 마스킹 필요 판단 |
| 비용·운영 | 우리가 읽으면 소스에 부하? | API rate limit / DB 읽기 replica |

**함정**: ① 소스에서 데이터를 *고치려* 든다(원본 보존 원칙 위반). ② 정적 파일이라 "변경 통보" 질문을 건너뜀 → 조용한 컬럼 변경이 1위 장애. 통보자 없으면 **스키마 스냅샷 diff**로 대체.

## 수집 (Ingestion)

> 소스에서 가져와 저장 직전까지 안전하게 운반. 형태 변환은 최소.

| 차원 | 이 단계의 질문 | 답이 갈리는 예 |
|---|---|---|
| 기능 | 원본 그대로 vs 최소 파싱? 토픽 분리? | 노출/클릭/전환 3토픽(SLA·볼륨 상이) |
| 성능·확장 | **배치 vs 스트림**? 푸시/풀/폴? 처리량(peak/avg)? | 신선도 1h→배치, 초단위→스트림 / 평균 12 EPS·peak 1.2k |
| 신뢰·복구 | **전달보장**(at-least/exactly-once)? 순서? 백프레셔? | 유실불가→at-least-once + 하류 멱등 / 순서는 파티션 키로 |
| 데이터품질 | 수집 시점 검증? 깨진 행 격리? | schema-on-read(Bronze 느슨) vs 검증 거절 |
| 보안·거버넌스 | 전송 암호화·인증? PII 즉시 처리? | TLS, uid 수신 즉시 해시 |
| 비용·운영 | 브로커 파티션·복제수? 마이크로배치 윈도우? | 자주 flush(신선도) ↔ small-file |

**함정**: ① "순서는 당연히 지켜지겠지" → 키 없으면 클릭이 노출보다 먼저 박힘. ② 즉시 flush → small-file 수만 개. 버퍼링 후 롤링.

## 저장 (Storage)

> 놓는 곳. 수집·변환·서빙을 떠받친다. 라벨(파티션)이 엉망이면 쿼리가 전체를 스캔.

| 차원 | 이 단계의 질문 | 답이 갈리는 예 |
|---|---|---|
| 기능 | 원본 append / 갱신 MERGE / 집계? 계층(medallion)? | Bronze append + Silver MERGE + Gold 집계 |
| 성능·확장 | 파티셔닝 키·입도? 파일 크기? 읽기/쓰기 속도? | event_date 일 단위(쿼리 가지치기) / 128MB 타겟 |
| 신뢰·복구 | 내구성·복제? 스냅샷·타임트래블? | S3 11-nine / Iceberg 스냅샷 롤백 |
| 데이터품질 | 스키마 on-read/on-write? 진화? | Iceberg add-column 안전, drop은 하류 깨짐 |
| 보안·거버넌스 | 암호화·접근권한? 카탈로그? 보존(hot/cold)? | Glue 카탈로그, Bronze 30/90d→Glacier |
| 비용·운영 | 저장 단가? compaction/expire? small-file? | 파일 포맷/압축(ZSTD), 3종 유지보수 잡 |

**함정**: ① "S3에 Parquet 떨구면 끝" → 테이블 포맷(Iceberg) 없으면 수정·타임트래블·동시쓰기 불가. ② late MERGE와 compaction이 같은 파티션 충돌 → 시간대 분리.

**핵심**: 파일 포맷(Parquet)은 *파일*, 테이블 포맷(Iceberg)은 *테이블*. 늦은 전환을 과거에 UPDATE해야 하면 Iceberg가 필수. **저장 출력 스키마=produced contract**(하류 Athena/BI가 의존).

## 변환 (Transformation)

> 가공. 정제(Silver)·집계(Gold). 핵심은 "늦게 온 데이터로 과거를 조용히 고치는" 멱등성.

| 차원 | 이 단계의 질문 | 답이 갈리는 예 |
|---|---|---|
| 기능 | 무슨 변환·비즈니스 규칙? 데이터 모델링? | CTR/CVR/ROI 집계 / 스타스키마 vs OBT |
| 성능·확장 | 배치 주기? 분산 vs 단일? SQL vs 코드? 증분 vs rebuild? | 15분→1시간(파티션 정합) / Spark 분산 |
| 신뢰·복구 | **멱등성**·재처리? late 반영 lookback? DAG 의존? | MERGE key 안정성 / 최근 7일 lookback |
| 데이터품질 | 결측·중복 규칙? 재실행 정합(2회 동일)? | `-1` 처리 / 재처리 후 행수·합계 동일 검증 |
| 보안·거버넌스 | 변환 중 리니지? PII 파생? | 컬럼 계보 추적 |
| 비용·운영 | 컴퓨트 비용·ROI? 스케줄·재시도? | rebuild 범위 vs 비용 / Airflow DAG |

**함정**: ① MERGE lookback 안 자르면 전체 테이블 스캔 → 갈수록 느려짐(파티션 프루닝 필수). ② "MERGE 썼으니 멱등" 착각 → 멱등성은 *안정 key + 결정적 변환*에서. 재실행 2회 합계 동일로 검증.

## 서빙 (Serving)

> 꺼내 쓰게 하기. 여기서 흐름이 뒤집힌다 (push→pull, 소비자가 요청). 사고가 처리량→지연·동시성으로.

| 차원 | 이 단계의 질문 | 답이 갈리는 예 |
|---|---|---|
| 기능 | 용도(분석 BI / ML / 역ETL)? 무슨 질문? | 캠페인×일자 KPI 대시보드 |
| 성능·확장 | **SQL vs API**? 사전집계 vs 원천? 동시성·지연? | Athena+Gold 사전집계(스캔↓) / 저지연 단건은 KV |
| 신뢰·복구 | 가용성? 타임아웃·폴백? | 워크그룹 동시쿼리 한도 |
| 데이터품질 | 신뢰·정합(시맨틱 레이어)? 채택? | "이 숫자=진실" 보장, 분석가가 실제 씀 |
| 보안·거버넌스 | 조회 권한? 컬럼/행 마스킹? | Lake Formation 광고주별 격리 |
| 비용·운영 | 쿼리당 스캔량·비용 상한? | 스캔 바이트 과금 → 파티션 프루닝 |

**함정**: ① small-file → WHERE 걸어도 안 빨라짐(compaction 필요). ② "WHERE 걸었으니 싸겠지" → 파티션 키 아닌 컬럼 필터는 여전히 풀스캔(스캔 바이트 과금).

**소비(Consumption)**: 최종 소비자가 **무슨 의사결정**을 하나 / 사람(대시보드) vs 기계(피드·ML) / 우리가 주는 계약(produced contract) — 이 출력은 누군가의 소스가 된다. 운영 가시성(위젯·health쿼리·알람)은 **비용·운영 차원**(DataOps)으로 다룬다.

## 언더커런트 (Undercurrents) — 전 단계 관통

위 표의 "보안·거버넌스"와 "비용·운영" 차원이 곧 이것들이다. 단계가 아니라 *모든 단계에 매번 함께* 묻는다 (Reis & Housley Ch.3):

| 언더커런트 | 매 단계에서 묻는 것 |
|---|---|
| **보안** | 최소 권한, 암호화(전송/저장), PII 분류·마스킹 |
| **데이터 관리(거버넌스)** | 카탈로그, 데이터 품질, 리니지, 규제(개인정보) |
| **DataOps** | 자동화(CI/CD), 관측성·모니터링(신선도·lag·알람), 사고 대응 |
| **데이터 아키텍처** | 트레이드오프, 확장성, 단순성, 가역성 |
| **오케스트레이션** | 의존 그래프·스케줄·백필·재시도 (Airflow) |
| **소프트웨어 엔지니어링** | 테스트, 코드 품질, IaC, 버전관리 |

## 한 장 요약

| 단계 | 정의 | 1순위로 걸리는 차원 |
|---|---|---|
| 생성 | 소스가 만듦 | 데이터품질(스키마·drift)·보안(PII) |
| 수집 | 가져오기 | 성능(배치/스트림)·신뢰(전달보장) |
| 저장 | 놓기 | 성능(파티셔닝)·비용(포맷·보존) |
| 변환 | 가공 | 데이터품질(멱등)·성능(주기) |
| 서빙 | 꺼내 쓰게 | 성능(사전집계·동시성)·비용(스캔) |
