# 데이터 플랫폼 설계 방법론

> **기반**: *Data Engineering Lifecycle* (Reis & Housley, *Fundamentals of Data Engineering*, O'Reilly) + *Well-Architected Framework* 품질 기둥 (AWS/Azure/GCP 공통).
> **핵심 명제**: 어떤 플랫폼이든 **질문은 불변**, **답(기술 선택)은 도메인·환경마다 다름**.
> **왜 이 문서**: 완벽히 설명 가능한 인프라/플랫폼 — 각 결정을 요구사항까지 되짚을 수 있는 상태.

---

## 0. 두 축으로 본다 (이게 전부)

구체 질문을 단계마다 나열하면 끝이 없다. 그래서 **질문을 두 축의 교차로** 본다:

```
질문 = 단계(WHERE, 어디서) × 요구사항 차원(WHY, 무엇을 위해)
```

- **단계** = 데이터가 거치는 곳. 불변.
- **요구사항 차원** = 그 질문의 *목적*. 모든 질문은 하나의 차원으로 올라간다.
- **답** = 기술·아키텍처. 환경에 따라 달라진다.
- **설명가능** = 어떤 결정이든 "어느 단계의, 어느 차원 요구 때문에, 왜 이 기술" 로 되짚을 수 있음.

---

## 1. 가로축 — 요구사항 차원 (질문을 묶는 것, WHY)

뿌리는 **요구사항**, 가지치기하면 6개 차원. **모든 구체 질문은 이 중 하나가 목적**이다.

```
요구사항 (왜 만드나 — 목적)
├─ 기능 (Functional) ........... 무슨 데이터·지표·비즈니스 로직
└─ 품질 (Non-functional) — Well-Architected 기둥 + 데이터 특화
   ├─ 성능·확장 (Performance) ... 얼마나 빠르게/많이 — 신선도·지연·처리량·규모
   ├─ 신뢰·복구 (Reliability) ... 안 끊기고 안 잃고 — 가용성·내구성·전달보장·RTO/RPO
   ├─ 데이터품질 (Data Quality) . 맞는 데이터 — 정확·정합·중복·멱등·스키마
   ├─ 보안·거버넌스 (Security) .. 안전·규정 — PII·접근제어·암호화·리니지·규제
   └─ 비용·운영 (Cost & Ops) .... 싸고 운영가능 — TCO·관측성·자동화·유지보수
```

> 쓰는 법: 각 단계에서 이 6개를 훑으면, *그 단계에서 실제로 load-bearing한 질문만* 떠오른다. 끝없이 나열하지 않는다.

---

## 2. 세로축 — 단계 (Data Engineering Lifecycle, WHERE)

```
[소스 시스템] → 생성 ── 수집 ── 저장 ── 변환 ── 서빙 ──→ [소비: 분석/ML/역ETL]
             Generation Ingestion Storage Transform Serving
```

- 생성(Generation): 소스가 데이터를 만드는 곳 / 수집(Ingestion): 가져오기 / 저장(Storage): 놓기 / 변환(Transformation): 가공 / 서빙(Serving): 꺼내 쓰게 하기 → 소비.
- 단계별 상세는 [`DESIGN_STAGES.md`](DESIGN_STAGES.md).

---

## 3. 격자 — 단계 × 차원 = 질문

각 칸이 "이 단계에서 이 차원을 어떻게 만족시키나". **이 격자가 질문 세트다** (구체 예시·답은 STAGES).

| 차원 ↓ \ 단계 → | 수집 | 저장 | 변환 | 서빙 |
|---|---|---|---|---|
| **기능** | 무슨 이벤트를 받나 | 무슨 계층/테이블 | 무슨 지표로 가공 | 무슨 질문에 답 |
| **성능·확장** | 배치/스트림·처리량 | 파티셔닝·파일크기 | 배치주기·분산 | 사전집계·동시성·스캔량 |
| **신뢰·복구** | 전달보장·백프레셔 | 내구성·복제 | 재처리·DAG 재시도 | 가용성·타임아웃 |
| **데이터품질** | 수집 시 검증 | 스키마 진화 | 멱등 MERGE·정합 | 시맨틱·신뢰(채택) |
| **보안·거버넌스** | 인증·PII | 암호화·카탈로그·보존 | 리니지 | 접근제어·마스킹 |
| **비용·운영** | 인프라·모니터링 | 저장비용·보존 | 컴퓨트·스케줄 | 쿼리비용·알람 |

> "왜 수집에 '배치/스트림'을 묻나?" → **성능·확장(신선도)** 칸이라서. 질문의 목적이 칸 위치로 드러난다.

---

## 4. 답이 갈리는 축 — 기술 선택 (환경, HOW)

같은 질문·같은 차원이라도, 환경이 답을 가른다 (책 4장 *Choosing Technologies*):

- **팀 규모·역량** — 운영 가능한 복잡도인가
- **속도 vs 견고함** — time to market
- **비용** — TCO + 기회비용
- **Build vs Buy**, **모놀리식 vs 모듈러**, **서버리스 vs 서버**
- **상호운용성** — 기존 스택과 붙나
- **현재 규모 + 미래 규모** (지금 1M/day → 100x)
- **위치** — 클라우드 / 온프렘 / 하이브리드

> 순서: **차원(왜) → 단계의 질문(무엇) → 요구 수치 → 환경 축 → 기술(답).** 기술을 먼저 고르고 정당화하지 않는다.

---

## 5. 설명가능성 점검 (결정마다)

- **어느 차원 때문인가?** ↑ — 격자에서 이 결정이 속한 차원·요구를 가리킨다. (못 가리키면 목적 없는 결정)
- **왜 다른 기술은 아닌가?** ★ — 기각한 대안 + 이유. (설명가능성의 핵심)
- **빼면 뭐가 깨지나 / 무엇을 희생했나?** ↔ — 하류 영향과 trade-off.

> 최종 검증: **문서 덮고 5분 구두 설명.** 통과 = 설명 가능한 플랫폼.
