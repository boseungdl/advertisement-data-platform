> 작성일: 2026-05-09
> 단계: 1단계 기획(Planning) 완료본
> 워크플로우 가이드 §2 PRD 양식 + 정형 템플릿 3종(규모/NFR/성공기준) 적용

## 1. 문제 정의 (한 줄)

광고 캠페인 매니저가 700+ 캠페인의 성과를 매일 캠페인 단위로 추적하면서, 데이터 파이프라인 운영자가 5분 안에 데이터 신선도와 헬스를 점검할 수 있는 단일 대시보드.

---

## 2. 페르소나

### P1. 캠페인 매니저 — "민혁" (28세, 광고대행사 운영팀)

- **역할**: 700+ 캠페인을 매일 모니터링하고 광고비 배분 결정
- **사용 맥락**: 매일 오전 9시 출근 후 첫 30분
- **페인포인트**
  - 어제 클릭 대비 오늘 들어온 전환을 매번 SQL로 다시 짜야 함
  - "어제 광고 효과 없네" 판단했는데 5일 후 늦은 전환이 들어와 결과가 뒤집힘
  - 어트리뷰션 방식(Last-click vs Even-credit) 결과가 다른데 둘을 비교 못 함
- **니즈**
  - 캠페인별 ROI/CTR/CVR 자동 계산
  - 늦게 도착한 전환이 어제·그제 표에 자동 반영
  - 두 어트리뷰션 결과 한 화면에 비교
- **성공 기준**: 매일 보고 준비 시간 1시간 → 10분

> 용어 풀이
> - **CTR**(Click-Through Rate) = 노출 대비 클릭 비율
> - **CVR**(Conversion Rate) = 클릭 대비 전환 비율
> - **ROI**(Return on Investment) = 광고비 대비 수익
> - **어트리뷰션** = "이 한 건의 전환이 어떤 광고 덕분이지?"를 결정하는 규칙

### P2. 파이프라인 운영자 — "지수" (31세, 데이터플랫폼팀)

- **역할**: 회사 전체 데이터 파이프라인 운영 책임
- **사용 맥락**: 매일 오전 출근 후 5분 헬스체크 + 알람 발생 시
- **페인포인트**
  - 데이터가 늦게 들어오는지 매번 SQL을 직접 짬
  - Iceberg 작은 파일이 누적되는데 정리를 누가 언제 했는지 모름
  - 새벽 Spark Job이 죽으면 어디부터 봐야 할지 RUNBOOK이 없음
- **니즈**
  - 신선도 / 행 수 / 파일 수 / 스냅샷 수가 한 화면에
  - 컴팩션·옛 스냅샷·고아 파일 자동 청소
  - 헬스 쿼리가 대시보드 패널로 박혀 있음
- **성공 기준**: 매일 헬스체크 5분 안에 끝, 야간 호출 시 10분 안에 진단

## 3. 핵심 기능 (MoSCoW)

> **MoSCoW** = Must / Should / Could / Won't 4단계 우선순위

### Must (반드시, MVP)

- **M1.** Criteo CSV → Kafka → Bronze(raw 파일) → Silver(Iceberg `processed_events`) → Gold(Iceberg `campaign_summary`) 3계층 파이프라인
- **M2.** 늦게 도착하는 전환에 대한 MERGE 멱등성 (같은 윈도우 두 번 돌려도 결과 동일)
- **M3.** Iceberg 매니지먼트 3종 자동화 (Compaction / Expire Snapshots / Orphan Cleanup) — Airflow DAG
- **M4.** 운영 헬스 쿼리 5~10개 (Iceberg 메타 테이블 기반)
- **M5.** 대시보드 탭 2개 — 비즈니스(캠페인 KPI) / 운영(헬스)

### Should (있으면 좋음)

- **S1.** Last-click vs Even-credit 어트리뷰션 결과 컬럼 둘 다
- **S2.** 100x 스케일 시나리오 ADR (구현 X, 설계만)
- **S3.** 장애 시나리오 RUNBOOK 1건 (Spark OOM 재시작)
- **S4.** README의 final-project 평가 답변 9개 섹션 완성

### Could (여유 있을 때)

- **C1.** 봇 클릭 필터링 (Silver 단계 룰 기반)
- **C2.** 백필 절차 자동화 스크립트

### Won't (이번 범위 X)

- ML 모델 (CTR 예측 등)
- RTB 입찰 시뮬레이션
- 멀티 광고주 분리 (Criteo에 광고주 컬럼 없음, 캠페인 단위로 충분)

## 4. 정형 템플릿 ① — 데이터 규모와 패턴

### 현재 규모 (가정)

- 일 ~100만 이벤트 (Criteo 16.4M / 30일 ≈ 일 547K → 라운드업)
- 시간당 ~4만, 초당 ~12
- 평균 이벤트 크기 ~1KB → 일 ~1GB raw / 월 ~30GB / 연 ~365GB

### 트래픽 패턴

- 피크: 평소의 2~3배 (12~18시 + 21~23시)
- 주말: 1.5배
- 계절성: 11월(블랙프라이데이/광군제), 12월(연말) 1.5배

### 성장 전망

- 단기(6개월): 10x → 일 1,000만 이벤트
- 장기(1~2년): 100x → 일 1억 이벤트 (일 ~100GB)
- → 파티셔닝 + Iceberg hidden partitioning + 카탈로그 분산 결정의 근거

## 5. 정형 템플릿 ② — 비기능 요구사항 (NFR)

> **NFR**(Non-Functional Requirement) = 기능 외에 시스템이 만족해야 할 품질 조건

| 항목 | 목표 |
|---|---|
| 가용성 SLA | 99.5% (월 다운타임 3.6시간 이내) |
| Kafka → Bronze 지연 | P95 1분 |
| Bronze → Silver MERGE | P95 15분 (5분 주기 배치) |
| Silver → Gold 집계 | P95 1시간 |
| 처리량 (평소) | 12 events/s |
| 처리량 (피크) | 30 events/s |
| 데이터 보존 (핫) | Iceberg 활성 스냅샷 7일 |
| 데이터 보존 (콜드) | Iceberg 옛 스냅샷 30일 후 Expire |
| 데이터 보존 (raw) | 90일 (백필 안전 윈도우) |
| RTO | 1시간 |
| RPO | 15분 |
| PII 포함 | N (Criteo 익명화) |
| 외부 노출 엔드포인트 | N |
| 데이터 라이센스 | CC BY-NC-SA 4.0 (Criteo) |
| AWS 월 비용 상한 | 5만원 이내 |

> **RTO**(Recovery Time Objective) = 장애 발생 후 서비스 복구까지 걸려도 되는 시간
> **RPO**(Recovery Point Objective) = 허용 가능한 데이터 손실 시간

## 6. 정형 템플릿 ③ — 성공 기준

### 기능적 성공

- 일 100만 이벤트 처리 가능
- 같은 윈도우 두 번 MERGE → 결과 동일 (멱등성 자동 테스트 통과)
- 헬스 쿼리 5~10개 모두 Athena에서 1초 이내 응답

### 운영 성공

- 매니지먼트 DAG 일 1회 자동 실행, 실패율 0%
- 컴팩션 직후 파티션당 파일 평균 5개 이내
- 30일 초과 옛 스냅샷 자동 만료

### 비즈니스 성공 (페르소나 기준)

- P1 민혁: 캠페인 KPI 일일 보고 시간 1시간 → 10분
- P2 지수: 헬스체크 5분 이내 완료, 한 화면에 모든 운영 메트릭
