# 데이터 플랫폼 설계 가이드

> **핵심 명제**: 어떤 플랫폼이든 **질문(절차·고려사항)은 불변**, **답(기술 선택)은 도메인·환경마다 다름**.
> **목표**: 각 결정을 요구사항까지 되짚을 수 있는 = **완벽히 설명 가능한** 인프라/플랫폼.

---

## 큰 틀 — 데이터 모델링 표준 5단계

설계는 아래 표준 절차를 따른다:

```
1. 요구사항 분석 → 2. 개념 모델링 → 3. 논리 모델링 → 4. 물리 모델링 → 5. 운영·유지보수
   Requirements     Conceptual       Logical          Physical         Maintenance
```

- **앞 단계(1·2)는 특정 플랫폼(RDB/레이크하우스) 선택에 중립적** — 도메인/비즈니스 중심이라 어느 플랫폼이든 대체로 동일하다. (단 소스 접근·플랫폼 필요성 판단은 실현 가능성을 일부 건드린다)
- **뒤로 갈수록 플랫폼별로 갈린다** — 정규화 테이블(RDB) vs 메달리온·스트리밍·파일포맷(레이크하우스).
- 각 단계는 **무엇을 / 실무 활동·질문 / 산출물**로 정리한다.

**근거(Sources)**
- 데이터 모델링 5단계·개념 모델: [Secoda — Data Modeling Documentation](https://www.secoda.co/glossary/data-modeling-documentation) · [Airbyte — Conceptual Data Model](https://airbyte.com/data-engineering-resources/conceptual-data-model) · [EWSolutions — Conceptual Data Modeling Starts with Business Use Cases](https://www.ewsolutions.com/conceptual-data-modeling-starts-with-business-use-cases/)

## 이 방법론의 유효 범위 & AI 사용법 (냉정하게)

> 4인 전문가 리뷰 반영. "Step 1만 정밀하면 AI가 나머지를 뽑아준다"는 유혹을 정면으로 차단한다.

- **자동 도출은 없다.** "Step 1 정밀 → AI가 2~5 생성 → 사람은 검증만" 은 환상이다(MDA·워터폴이 반복 실패한 그것). Step 1 → 2~5는 결정론적 *도출*이 아니라 **제약 전파** — Step 1은 해공간을 좁히고, AI는 그 안에서 *한 후보*를 낼 뿐. 남은 자유도(gap)는 **사람이 결정**한다. *"도출처럼 보이는 것은 검증 안 된 보간(interpolation)이다."*
- **AI의 역할** = 초안 생성 + **반례 제시(red-team)** + 누락 지적 *가속기*. 사람 = 모든 결정을 내리고 방어. (LLM의 진짜 강점은 생성보다 **반례 생성** — 단방향 생성기로만 쓰면 강점을 버린다.)
- **단계마다 신뢰도가 급락한다:**
  - **2 개념 / 3 논리**: AI 생성 + 정적 검증이 대체로 성립 (단 3은 판단·ADR 비중↑)
  - **4 물리 / 5 운영**: 파티션·비용·SLA·알람 임계는 요구사항이 아니라 **실측**에서 나온다 → "AI 생성 → 읽고 검증"이 아니라 **"초안 → 측정 → 튜닝 루프"**.
- **검증 ≠ 항상 쉬움.** 그럴듯한 오답(잘못된 어트리뷰션 등)은 파이프라인이 완벽히 돌아도 숫자만 조용히 틀린다. 그리고 *검증에 필요한 전문성 = 생성에 필요한 전문성* → **2~5를 이해 못 하면 검증도 못 한다.**
- **흐름은 → 가 아니라 ⇄.** 뒤 단계 발견이 앞 단계를 역류시켜 고치는 건 정상(실패 아님).
- **각 단계 완성기준은 3분류로 태그한다:**
  - **[체크]** 정적 판정 (fact-to-fact 없나, grain 닫혔나) — 읽고 O/X
  - **[결정+ADR]** 여러 유효안 — 사람이 고르고 ADR 기록
  - **[측정]** 돌려봐야 앎 (파티션 효율·비용·임계) — 측정 절차 + 통과 임계
- **자기기만 경보**: 페르소나·`[가정]`·single-LLM role-play는 "출제자=채점자"가 되기 쉽다. 핵심 결정에 걸린 `[가정]`은 다음 단계로 넘기지 말고, 최소 한 번은 LLM에게 **red-team**(이 설계가 터지는 시나리오는?)을 돌려 모순을 강제 노출한다.

## 1단계 — 요구사항 분석 (Requirements Analysis)

### 목적
도메인을 이해해 *"무엇이 필요한가"* 를 요구사항으로 옮긴다. **입력 = 도메인 이해 / 출력 = 요구사항 명세.** (데이터 구조를 그리는 건 2단계)

> **오해 주의**: 도메인을 *먼저 다 알아야* 하는 게 아니다. 아래 질문들이 곧 도메인을 알아내는 도구고, 이벤트(예: 노출→클릭→전환)는 *"어떤 프로세스가 데이터를 만드나?"* 의 **답으로 발견**된다. 답은 지어내는 게 아니라 도메인에서 나온다.

### ① 시작 전 — 있어야 하는 것 (전제)
데이터 플랫폼은 데이터를 *소비·분석*하는 레이어라, 데이터를 *만드는 것*이 먼저 있어야 한다.

```
서비스 운영(데이터 생성) → 데이터 쌓임 → 분석 니즈 발생 → 플랫폼 구축
```

- **소스(데이터 생산자)** — 운영 서비스+DB / 로그 / 외부 데이터. 최소 하나. *(연습이면 공개 데이터셋이 소스를 대신)*
- **분석 니즈** — 왜 짓나 (이 단계에서 확정)
- **데이터 접근 경로** — DB 읽기 권한 / API 키 / 로그 수집
- **인프라 기반** — 저장·컴퓨트·오케스트레이션 환경 (클라우드 등)

**+ 페르소나 (실무가 아닐 때 "조직 맥락"을 대체)**
비기능의 일부(예산·목표 SLA 수준·컴플라이언스)는 도메인이 아니라 *조직/기업*이 정한다. 연습 프로젝트엔 회사가 없으니 → **가정된 기업 프로필(페르소나)** 을 한 번 정해 그 자리를 채운다.
- 정할 것: 회사 단계(스타트업/대기업) · 비용 posture · 목표 SLA 수준 · 컴플라이언스 환경 · 팀 규모
- 예: *"중견 애드테크 스타트업 / 비용 민감 / 가용성 99.5% 목표 / 개인정보 규제 있음 / 데이터팀 3명"*
- 이 페르소나 기준으로 조직-유래 결정을 **일관되게** 내림 → "이 페르소나라서"로 설명 가능 (= 흩어진 `[가정]`의 출처).

### ② 어떻게 — 질문(프롬프트) → 각각의 결과
혼자면 각 프롬프트를 LLM에게 role-play 시키거나, 리서치·인터뷰 스크립트로 쓴다.
- 혼자 구한 답엔 `[가정]` + **"무엇으로 반증되나"** 를 함께 적는다 (반증 불가능한 가정 = 자기기만).
- **핵심 결정(grain·1건 정의·어트리뷰션·신선도·규모·PII)에 걸린 `[가정]`은 다음 단계로 넘기지 않는다** — 확정 후 진행.
- **데이터셋 방화벽**: 연습용 공개 데이터셋을 써도 **스키마를 보기 전에 위 도메인 답을 먼저 고정**한다. 스키마는 검증용이지 도출용이 아니다(역유추 금지).

**1. 비즈니스 목표 — 역할: 프로덕트 오너(PO)**
> "당신은 [서비스명]의 프로덕트 오너입니다. 이 데이터 프로젝트로 이루려는 **비즈니스 목표**가 무엇인가요? 그 성과를 어떤 지표(OKR/KPI)로 측정하나요? 이 목표가 달성되지 않으면 회사는 무엇을 잃나요?"
>
> **→ 얻어야 하는 결과**: 비즈니스 목표(1~2문장) + 측정 지표(이름+목표치) + 미달 시 임팩트(우선순위 근거)
> *예: 목표=성과 높은 캠페인에 예산 재배분 / 지표=ROAS / 임팩트=예산 낭비*

**2. 답할 질문 — 역할: 실제 사용자(분석가)**
> "당신은 이 데이터를 매일 쓸 분석가입니다. 이 데이터로 **어떤 질문에 답하고 싶으신가요?** 그 답을 알면 무슨 결정을 내리나요? 구체적 질문 3~5개를 들어주세요."
>
> **→ 얻어야 하는 결과**: 답해야 할 질문 목록(3~5개) + 각 질문 뒤의 결정
> *예: "어느 캠페인이 전환당 비용(CPA)이 가장 낮나?" → 그 캠페인에 예산 집중*

**3. 프로세스·이벤트 — 역할: 도메인 전문가**
> "당신은 [광고] 도메인 전문가입니다. 이 데이터를 만드는 **프로세스를 처음부터 끝까지** 설명해주세요.
> (a) **이벤트를 순서대로** + 각 시점·트리거 + **시간 차이(늦게 오는 것)의 분포·상한**(P50/P95/최대 며칠?),
> (b) 이벤트/엔티티 간 **개수 관계**(노출 1건에 클릭 N개? 전환 없는 클릭?),
> (c) **중복·불가능한 조합·각 지표 정상범위**(CTR 상한 등),
> (d) **엔티티 속성이 시간에 바뀌고 과거 값이 중요한지 + 그 과거 값이 *어떤 결정에* 쓰이는지**(예: 예산을 거래 시점 기준으로 봐야 함),
> (e) 각 핵심 이벤트에서 **'1건'의 경계**는? (전환 1건 = 주문? 결제? 라인아이템? 같은 사용자가 두 번이면 1건? 2건?),
> (f) 결과 이벤트(전환)를 원인 이벤트(클릭/노출)에 **어떻게 귀속(어트리뷰션)** 하나? (마지막 접점? 윈도우 며칠? 뷰스루 포함?)"
>
> **→ 얻어야 하는 결과**:
> - 이벤트 목록(순서)+시점/트리거+**딜레이 분포·상한** *(← 데이터로 넘어가는 다리 / 재집계 윈도우 근거)*
> - 개수 관계(카디널리티), 유효성 규칙(중복·불가능조합·정상범위)
> - 시간 변화 + **그 근거(어떤 결정에 쓰나)** *(← SCD 판단 근거)*
> - **'1건'의 정의(grain 경계)** *(← 2단계 grain 확정의 입력)*
> - **어트리뷰션 규칙 + 윈도우** *(← 전환 팩트·degenerate dim 설계의 입력)*
> *예: 노출(즉시)→클릭(초~분)→전환(P95 3일·최대 30일) / 1노출:N클릭, 전환없는 클릭 존재 / CTR≤100% / 예산은 시점별 이력 필요(집행 리포트) / 전환 1건=결제 1회 / 어트리뷰션=last-click 7일 윈도우*

**4. 소스 & 규모 — 역할: 백엔드 개발자/DBA**
> "당신은 이 서비스의 백엔드 개발자입니다. 이 데이터는 **어디서**(시스템·테이블·로그) 생성되나요? **어떤 형태**(스키마)이고 한 건을 식별하는 **키**는? **규모**는요 — 하루 몇 건, 건당 크기, 피크/평균, 증가율, 총 보존량? **개인정보/민감 필드**(user_id·디바이스·IP)가 있나요? 우리가 **접근**할 수 있나요?"
>
> **→ 얻어야 하는 결과**: 소스 목록 + 형태(스키마 개요)·식별 키 + **규모(건수/일·크기·피크비·증가율·총량)** + **PII/민감 필드** + 접근 가능 여부
> *예: 광고서버 로그(JSON, 실시간, event_id) / 16M행·30일·건당~1KB / uid=PII→해시 / 연습=Criteo CSV*
> *단, 여기선 컬럼을 설계하지 않는다 — 접근성 확인 + 식별자·규모·grain 힌트 채취까지.*

**5. 신선도 — 역할: 의사결정자**
> "당신은 이 데이터를 보고 결정하는 사람입니다. 데이터가 **얼마나 최신이어야** 결정에 쓸 수 있나요? 하루 지난 것도 괜찮나요, 실시간이어야 하나요? 얼마나 늦으면 쓸모없어지나요?"
>
> **→ 얻어야 하는 결과**: 신선도 요구(숫자: 실시간/분/시간/일) + 허용 한계
> *예: 예산은 하루 단위 결정 → 신선도 1시간이면 충분*

**6. 소비자 — 역할: PO**
> "이 데이터를 최종적으로 **누가** 사용하나요? **사람**이 대시보드로 보나요, 아니면 **다른 시스템·ML 모델**이 받아가나요? 소비자가 여럿이면 각각 어떻게 쓰나요?"
>
> **→ 얻어야 하는 결과**: 소비자 목록 + 유형(사람/기계) + 각 접근 형태(대시보드/API/피드/ML)
> *예: 광고 분석가(사람·대시보드) + 향후 입찰 ML(기계·feature)*

**7. 지표 정의 — 역할: 분석가**
> "각 핵심 지표의 **정확한 계산식**은? 분자·분모의 정의(중복 제외? 봇·내부 트래픽 제외? gross vs net)는요?"
>
> **→ 얻어야 하는 결과**: 지표 정의 사전(계산식 + 분모/분자 정의) *(← measure vs 팩트 구분, 품질 검증의 입력)*
> *예: CTR = 클릭 / **unique** 노출(봇 제외) / CVR = 전환 / 클릭(어트리뷰션 윈도우 내)*

### ③ 결과 종합 — 요구사항 명세 (하나의 사슬)
질문은 선형(1→6)으로 하지만 종합하면 **트리**다 — 목표에서 갈라져 소스·신선도·소비자가 병렬 제약이 된다:

```
① 목표(왜) → ② 답할 질문(뭘 알아야) → ③ 이벤트/데이터(무슨 데이터 필요)
                                        → ④ 소스(어디서) + ⑤ 신선도(얼마나 최신) + ⑥ 소비자(누가)
```

목표에서 출발해 "무엇을 알고 싶나 → 무슨 데이터 → 어디서·얼마나최신·누가"로 좁혀진다. 이 사슬 전체가 **요구사항 명세**.

**+ 2단계(개념 모델링)로 넘길 씨앗을 여기서 표시** (모델을 그리진 않되, 재료를 밑줄):
- **분석 grain** = "무엇 1건 = 1행"인가 (질문 2·3·5 교차로 도출. 예: 캠페인×일자 집계 vs 원자 이벤트)
- **엔티티 후보** = 질문에 나온 명사 (캠페인·광고·사용자·노출·클릭·전환)
- **관계 후보** = 프로세스 동사·개수 관계 (사용자가 광고를 클릭 / 1노출:N클릭)
- **키 후보** = 각 이벤트를 유일 식별하는 자연 키 (질문 3·4)
- **'1건' 정의 · 어트리뷰션 규칙 · 지표 정의 사전 · late-arrival 상한** (질문 3e·3f·7 — grain·SCD·품질의 근거)

그리고 별도로 → **"데이터 플랫폼이 필요한가?" 판단** (아래 곁가지).

### ③-b 비기능(품질·신뢰·가용성·비용) 도출
비기능은 따로 묻는 항목이 아니라 **도출**한다 — 뿌리는 도메인, 일부는 페르소나.

- **도메인 사실 → 비기능** (대부분 여기서):

| 도메인 사실 (불변 범주, 괄호=광고 예) | → 도출되는 비기능 |
|---|---|
| 금전 직결 이벤트 *(전환)* | 신뢰성·가용성 높게 (돈 이벤트 등급화) |
| 고볼륨 이벤트 *(노출 대량)* | 저비용 저장·처리 |
| 지연 도착 이벤트 *(전환 딜레이)* | 정확도 (과거 갱신·멱등) |
| 값 제약 있는 지표 *(CTR≤100%)* | 데이터 품질 검증 규칙 |

- **페르소나 → 조직-유래 비기능** (도메인에서 안 나오는 것): 예산 상한 · 목표 SLA 수준(99.5% vs 99.99%) · 컴플라이언스.
- **경계 주의**: 여기선 *"어떤 비기능이 중요한가 + 목표 수준"* 까지만. 구체 실현(파티셔닝·멱등 MERGE·전달보장·배치 주기)은 **뒤 단계** (요구는 1단계 / 해결은 뒤).

### ④ 완성 기준 — 이렇게 되어 있어야 한다
- 목표가 **측정 가능한 지표**로 표현됨
- 이벤트가 **순서·타이밍**(늦은 도착 등) + **개수 관계·유효성 규칙·시간 변화(SCD)** 까지 파악됨
- **규모가 숫자로**(건수/일·크기·증가율) 정해짐 — "플랫폼 필요?" 판단의 근거
- 소스가 **실제 접근 가능**한지 확인됨 + **PII/민감 필드 식별됨**
- 신선도가 **숫자**로 정해짐
- 소비자·접근 형태가 명확
- **분석 grain**("무엇 1건=1행") 식별됨 + **엔티티·관계·키 후보** 표시됨 *(2단계 입력)*
- **'1건'의 정의 · 어트리뷰션 규칙 · 지표 계산식**이 명시됨 *(암묵지 캡처 — AI가 지어내지 못하게)*
- 모든 비기능·결정이 **도메인 사실/페르소나로 역추적 가능한 형태**(③-b 같은 매핑)로 기록됨 *(설명가능성)*
- 모든 `[가정]`에 **ID + 반증 방법**이 있고, 그 가정을 소비한 2단계 결정에 **역참조 가능**
- "플랫폼 필요?"에 **근거 있는 답**

→ 이 상태면 **2단계(개념 모델링)의 입력으로 충분**하다.

### 곁가지 — "데이터 플랫폼이 필요한가?" 판단
종합된 요구를 보고 **딱 하나**를 묻는다:

> **"이 분석을 운영 DB + 간단한 BI 툴로 감당할 수 있나?"**

- **감당 됨** → 플랫폼 불필요 (RDB + BI로 끝)
- **감당 안 됨** → 플랫폼 필요. *안 되게 만드는 이유* = {대용량 · 실시간(스트리밍) · 다중 소스 통합 · 이력/ML} 중 하나 이상
- 판단 축은 **거래(OLTP)냐 분석(OLAP)이냐**지, 스트리밍 여부가 아니다.
- **필요한 만큼만**: 감당 되면 더 싼 방법(읽기 복제본·분석 전용 DB)부터. 무조건 플랫폼은 over-engineering.

## 2단계 — 개념 모델링 (Conceptual Modeling)

### 목적
1단계 씨앗을 **개념 모델(엔티티-관계 + 팩트/디멘전)** 으로 정리한다. = 도메인을 *데이터 구조*로 옮기는 첫 걸음. 컬럼·최종 스키마는 3단계.

> **ERD는 "이해용 스케치"지 최종 스키마가 아니다.** 정규화 ERD를 남기는 건 OLTP, 분석 플랫폼은 이 개념 모델을 스타 스키마로 변환한다 — 그 갈림은 **3단계**. 2단계는 어느 쪽이든 공통인 "엔티티·관계·팩트/디멘전"까지만.

### A. 생성 계약 (AI 입력→출력)
- **입력**(1단계 산출물): 엔티티 후보 · 관계/개수관계 · 키 후보 · grain 씨앗 · 이벤트 목록 · 유효성 규칙 · SCD 씨앗 · **'1건' 정의 · 어트리뷰션 규칙 · 지표 정의 · late-arrival 상한**.
- **출력**(개념 데이터 모델): 엔티티(+식별자) / 팩트·디멘전 분류(+conformed) / 각 팩트 grain 문장·팩트종류 / 관계(공유 디멘전) / 이력 필요 디멘전(SCD) 표식. **컬럼·타입 없음.**
- **프롬프트 골격**: *"입력으로 개념 모델을 생성하라. 각 분류·결정에 근거(1단계 어느 사실/질문)를 붙여라. 팩트끼리 직접 잇지 말고 공유 디멘전으로 연결하라. 여러 유효안(grain 원자 vs 집계 등)은 하나 고르지 말고 C(판단 gap)에 후보로 나열하라. 컬럼·타입은 만들지 마라."*

> **용어**: 모든 이벤트는 엔티티다. 그중 *측정 대상*이 **팩트** (엔티티 ⊃ 팩트).

**출력 예시 (광고)**
```
   [캠페인*]   [광고]   [사용자]   [날짜]        ← 디멘전 (공유 = conformed)
       \_________\_________/_________/
                     ↑ 세 팩트가 모두 참조
       《노출》      《클릭》      《전환》          ← 팩트 (이벤트)
[ ]=디멘전  《 》=팩트  * =이력 필요(SCD)
※ 팩트끼리 직접 연결 X — 공유 디멘전으로만.
※ 순서/개수(노출→클릭→전환, 1노출:N클릭, 전환없는 클릭)는 도메인 사실(주석).
※ 클릭→전환 귀속: 전환 팩트에 click_id = degenerate dim 후보.
(예시일 뿐 — 핀테크=계좌·거래·정산 / 이커머스=상품·주문·결제로 동일 절차)
```

### B. 강제 규칙 [체크] (pass/fail)
- **[정규칙]** 모든 이벤트/명사가 엔티티/속성으로 분류 + 각 엔티티에 **개념적 식별자**
- **[정규칙]** 팩트/디멘전 구분 + **conformed 디멘전(날짜 등)** 식별
- **[정규칙]** 각 팩트에 **grain 한 문장** + **팩트 종류**(트랜잭션/주기스냅샷/누적스냅샷)
- **[부정칙]** 팩트-투-팩트 **직접 연결 없음** (있으면 FAIL — 안티패턴)
- **[부정칙]** **컬럼·타입 등장 없음** (경계 침범 FAIL)
- **[정합]** 1단계 항목이 **하나도 안 빠지고** 반영 (누락 0 = 커버리지):
  - 각 팩트 grain이 1단계 **'1건 정의'와 일치**? 불일치 → gap
  - 어트리뷰션이 **degenerate dim으로 표현 + 윈도우가 1단계와 일치**?
  - SCD 표식 디멘전마다 **1단계 근거(질문 3d)** 링크? 근거 없는 SCD = FAIL

### C. 판단 gap [결정+ADR]
- **grain 원자 vs 집계** — 원자=재처리 유연·저장↑ / 집계=쿼리 빠름·되돌리기 어려움. 물을 것: 재집계 필요? 다운스트림? → ADR
- **전환 팩트 = 누적 스냅샷 vs 트랜잭션+재집계** — late-arrival 상한 기준. → ADR
- **click_id 표현 = degenerate dim vs 별도 브리지** — 어트리뷰션 복잡도. → ADR

### D. 측정 [측정]
- 개념 단계는 실측 대상 **거의 없음** (물리 성능·비용은 4단계). 대신 사고실험: *"대표 분석 질문 3개가 이 모델로 답 가능한가?"* — 안 되면 gap.

### E. 완성 기준
- **B 전부 pass** (정규칙 존재 + 부정칙 위반 0 + 정합 커버리지 누락 0)
- **C 전부 ADR로 닫힘**

→ 이 상태면 **3단계(논리 모델링)의 입력으로 충분**하다.

### F. 실패 루프 (⇄)
- B 정합 실패(1단계 항목 누락) → AI 재생성(누락 지적 재주입). 그 항목이 1단계에 **원래 없었으면 → 1단계 역류(보완)**
- C 결정 불가 → 1단계 제약 부족 → 역류
- 부정칙 위반(팩트-투-팩트 등) → 재생성
