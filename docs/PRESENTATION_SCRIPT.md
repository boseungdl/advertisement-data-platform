## §1. 도메인 정의 + 핵심 KPI 3개

### §1-A. 도메인 분야 (한 줄 카피)

**[화면]** README §1 도메인 분야 부분 보이도록 스크롤.

**오프닝 멘트**:
> "광고 도메인의 클릭, 노출, 전환 이벤트를 메달리온 아키텍처로 관리하는 데이터 레이크하우스에 대한 발표, 시작하겠습니다."

**호흡 가이드**:
- "광고 도메인의" — 한 박자 쉬고
- "클릭, 노출, 전환" — 세 단어를 끊어서 또박또박 (광고 데이터의 3대 이벤트라는 인상)
- "메달리온 아키텍처로 관리하는" — 한 호흡으로
- "데이터 레이크하우스" — 마무리 강조
- "발표, 시작하겠습니다" — 인사 톤

**보충 멘트 (필요 시)**:
> 🟢 데모와의 차이를 청중이 모를 수 있으니 한 줄로 — "강의 데모는 도메인이 정해지지 않은 generic Iceberg 메달리온이지만, 저는 광고 도메인으로 좁혀서 클릭·노출·전환의 본질을 다뤘습니다."

**예상 청중 반응**:
- 강사 / 현업 데이터 엔지니어 → "메달리온"·"레이크하우스" 단어로 강의 주제와 연결 즉시 인식
- 대표 / 비전공 → "광고 + 클릭/노출/전환" 단어로 대략 그림 잡힘
- 동기 → 여기서는 아직 차별화 안 보일 수 있음 → §1-C 또는 §4에서 풀어줌

### §1-B. 데이터 규모와 패턴

**[화면]** README §1 데이터 규모와 패턴 섹션 (5개 bullet).

**대본**:
> "데이터 규모와 패턴은 다섯 가지로 정의했습니다."
>
> "첫째, 현재는 일 100만 이벤트, 시간당 4만, 초당 12개의 작은 환경입니다."
>
> "둘째, 평균 이벤트 크기 1KB로 일 1GB raw, 한 달 30GB, 1년 365GB가 쌓입니다."
>
> "셋째, 트래픽 패턴은 광고 도메인에 맞춰 직접 정의했습니다. 피크 시간대 평소의 2~3배, 주말 1.5배. 시즌 스파이크는 §7에서 깊이 다루겠습니다."
>
> "넷째, 6개월 내 10배 성장 — 일 1,000만 이벤트가 현실적 목표입니다."
>
> "**다섯째, 1~2년 내 100배까지 — 일 1억 이벤트, 일 100GB.** 이 100배 가정이 이 프로젝트의 모든 아키텍처·스키마 결정을 통과시키는 기준이고, 뒤에 나올 §7 스케일 아웃 시나리오의 출발점입니다."

**호흡 가이드**:
- "다섯 가지로 정의했습니다" — 청중에게 다섯 항목 들을 마음의 준비
- 첫째~넷째는 빠르게 또박또박 (숫자 강조, 부수 정보는 빠르게)
- "**다섯째, 100배**" — 가장 강한 강조. 한 박자 쉬고
- "§7에서 다루겠습니다" — 다음 챕터 호기심 유도

**예상 청중 반응**:
- 강사 / 현업 → 100x 가정의 합리성 + "어디가 깨질까" 사고 시작
- 대표 → "지금 작지만 100배 갈 거다" 비즈니스 메시지
- 동기 → "왜 100x를 가정했나" 의문 → §7에서 답

**스피커 노트**:
- 시즌 스파이크(블프·광군제 3~5배)는 §1-B에서 풀지 말고 §7에서 깊이. 여기서는 "성장 가정 100x" 한 메시지에 집중
- 가이드 정형 템플릿 5항목을 모두 커버한 게 단단한 출발 — "도메인에 맞게 직접 정의"라는 표현이 광고 도메인 사고가 들어갔다는 시그널

### §1-C. 핵심 KPI 3개 — 데이터 엔지니어링 메인

**[화면]** README §1 핵심 KPI 3개 표.

**대본**:
> "이 프로젝트의 성공 기준은 세 가지 숫자로 정했습니다. 모두 데이터 엔지니어링 본질에 닿는 지표입니다."
>
> "첫째, **처리량** — 일 100만 이벤트 처리. 10배 성장하면 일 1,000만까지. 광고 도메인 표준 규모이고 강의 가정과 일치합니다."
>
> "둘째, **신선도** — 늦게 도착하는 전환을 P95 1시간 안에 Gold에 반영합니다. 즉, 100번 측정하면 95번은 한 시간 안에 어제 표가 자동 갱신됨을 보장합니다. **이게 광고 도메인의 본질**입니다 — 전환은 며칠 뒤 도착하기 때문에."
>
> "셋째, **멱등성** — 같은 윈도우를 두 번 돌려도 결과가 비트 단위로 동일합니다. 새벽에 잡이 죽었을 때 사람이 깨어나지 않아도 자동 복구되고, 3개월 백필도 안전합니다. 이 시스템의 **안전벨트**입니다."
>
> "어트리뷰션 모델을 어떻게 정의하느냐, ROI 임계값을 얼마로 잡느냐 — 이런 결정은 **분석가의 영역**입니다. 저는 **데이터 엔지니어**로서, 분석가가 그런 분석을 안전하게 수행할 수 있도록 **신선도와 멱등성을 보장하는 게 본업**입니다. 그래서 KPI 3개도 데이터 엔지니어링 본질에 집중합니다."

**호흡 가이드**:
- "세 가지 숫자" — 청중에게 "기능 나열이 아닌 측정 가능한 지표" 시그널
- 각 KPI 이름(**처리량 / 신선도 / 멱등성**) 강조 — 한 박자 쉬고
- "P95"는 풀어주기 — "100번 중 95번"
- "광고 도메인의 본질" — 강조 (이 프로젝트의 핵심 메시지)
- "안전벨트" 비유 — 비전공 청중도 즉각 이해

**예상 청중 반응**:
- 현업 데이터 엔지니어 → "멱등성"·"P95" 단어에 즉시 반응. 다음 슬라이드(§4 Iceberg)에서 답 기대
- 대표 / 비전공 → "안전벨트" 비유로 멱등성의 가치 즉각 이해
- 강사 → 강의 가정(일 100만) 일치 확인 + Phase 2 분리 사고 평가

**스피커 노트**:
- 처리량 / 신선도 / 멱등성 3개는 데이터 엔지니어가 가장 자신 있게 말할 수 있는 KPI 조합
- 분석 영역 KPI(어트리뷰션·ROI)는 Phase 2로 분리한 사고 자체가 발표 임팩트 — "메인 단단해진 후 분석 얹는다" 라는 운영 사고
- "P95 1시간 신선도" + "비트 단위 멱등성" → §4 Iceberg 가치(MERGE 멱등성·Snapshot Isolation)와 자연스럽게 연결

## §2. 전체 아키텍처

### §2-A. 왜 이 컴포넌트 조합인가 — 왜 레이크하우스인가

**[화면]** README §2 "왜 이 컴포넌트 조합인가" 페르소나 요구→컴포넌트 매핑 표 + 대안 비교 표.

**대본**:
> "이 아키텍처를 보면 청중이 가장 먼저 가질 의문이 있습니다 — **'광고 데이터 처리에 굳이 레이크하우스가 필요한가? Glue ETL로 변환해서 RDS에 넣어도 되지 않나?'**"
>
> "답은 페르소나 요구에 있습니다."
>
> "**캠페인 매니저는** 700개 이상 캠페인의 ROI와 어트리뷰션을 SQL로 ad-hoc 분석해야 하고, 며칠 뒤 도착한 전환으로 어제 표가 자동 갱신되어야 하고, 직관적 대시보드가 필요합니다."
>
> "**파이프라인 운영자는** 광고 이벤트가 stream으로 즉시 적재되어야 하고, 90일 raw를 비용 효율적으로 보존해야 하고, 백필·시간여행이 안전해야 하고, 5분 안에 헬스체크가 가능해야 합니다."
>
> "**Glue ETL만으로는 안 됩니다.** Glue는 batch 변환 도구이지 데이터 플랫폼이 아닙니다. 스트리밍이 없고, 시간여행이 없고, MERGE 멱등성이 없고, 운영 가시성도 별도로 만들어야 합니다. 페르소나 요구의 절반도 못 만족합니다."
>
> "**데이터 웨어하우스 단독도 부적합**입니다. raw를 90일 보존하면 비용이 폭증하고, 스트리밍 통합이 약합니다."
>
> "**그래서 레이크하우스 패턴**이 답입니다 — raw부터 분석까지 한 곳에, ACID와 시간여행이 보장되는 표 형식 위에서 두 페르소나 요구를 한 번에 만족시킵니다. 그래서 컴포넌트 조합이 — Kafka, Spark Streaming, S3, Iceberg, Athena, QuickSight, Airflow — 이렇게 결정됐습니다."

**호흡 가이드**:
- 첫 의문 — "굳이 레이크하우스가 필요한가?" 청중이 본인 머릿속에 이미 한 질문일 수 있음. 그 의문에 답하는 형태로 자연스럽게 시작
- 페르소나 두 명을 분리해서 — "캠페인 매니저는…", "파이프라인 운영자는…" 한 박자씩 쉬고
- "**Glue ETL만으로는 안 됩니다**" 강조 — 청중이 가장 흔히 떠올리는 대안에 정면으로 답
- "**그래서 레이크하우스 패턴**" — 결론, 한 박자 쉬고
- 마지막 컴포넌트 7개 나열은 빠르게 (이미 다이어그램에 있음, 보충일 뿐)

**예상 청중 반응**:
- 현업 데이터 엔지니어 → "Glue ETL 대안 비교"에 즉시 동의. "이 사람 운영 사고로 컴포넌트 선정했네" 인식
- 강사 → 강의 데모 컴포넌트와 같은 결과지만 **결정의 근거가 페르소나 요구에서 나온 점** 평가
- 대표 → "왜 굳이 이렇게 비싼/복잡한 구성?" 의문이 자연스럽게 풀림
- 동기 → "어떻게 컴포넌트 정해야 하는지" 한 사례로 받음

**스피커 노트**:
- 이 슬라이드의 메시지: **컴포넌트 선택은 페르소나 요구에서 시작해야 한다**. 트렌드 따라가는 게 아님
- "Glue ETL만으로는" 부분이 가장 강력 — 청중이 흔히 떠올리는 합리적 대안에 정면 답변. 발표자의 사고 깊이가 드러남
- 이 슬라이드 다음에 §2-B(메달리온 3계층 결정)가 자연스럽게 따라옴 — "레이크하우스 안에서 어떻게 쪼갰나"

### §2-B. 메달리온 3계층 결정 — 레이크하우스 안에서 어떻게 쪼갰나

**[화면]** README §2 "출발점 — 무엇을 고민했나" 표.

**대본**:
> "이 아키텍처를 구성할 때 가장 먼저 고민한 건 광고 도메인의 본질입니다 — **클릭은 오늘 일어나지만 전환은 며칠 뒤에 도착한다**. 어제 만든 캠페인 리포트가 오늘 도착한 전환 데이터로 자동 갱신되어야 하고, 그 갱신 과정에서 데이터가 망가지면 안 됩니다."
>
> "이 한 가지 고민에서 세 가지 결정이 나왔습니다."
>
> "**첫째, Bronze raw는 Iceberg가 아닌 plain Parquet으로 둡니다.** raw는 백필 안전선이라 절대 손상되면 안 되고, append-only로 쌓기만 하면 충분합니다. Iceberg의 MERGE나 스냅샷 같은 기능이 raw에선 가치가 없고, 오히려 관리 부담만 늘기 때문입니다."
>
> "**둘째, Silver는 Iceberg 오픈테이블포맷으로 결정했습니다.** 며칠 뒤 도착한 전환을 어제 표에 안전하게 갱신하려면 MERGE 멱등성이 필수인데, 이건 plain Parquet으로는 불가능합니다. **이게 이 프로젝트에서 Iceberg를 채택한 가장 큰 근거**입니다."
>
> "**셋째, Gold는 한 테이블이 아니라 두 종류로 분리했습니다.** 같은 정제 데이터지만 운영자가 보는 헬스 메트릭과 캠페인 매니저가 보는 비즈니스 KPI는 갱신 주기·집계 단위·소비자 페르소나가 모두 다릅니다. 한 테이블에 섞으면 어느 쪽도 만족시킬 수 없어 분리한 결정입니다."
>
> "흐름 자체는 단순합니다 — Kafka, Spark Streaming, Iceberg, Athena. 다만 **각 단계의 결정 근거가 광고 도메인 운영의 본질에 닿아 있다**는 게 이 프로젝트의 본질입니다."

**호흡 가이드**:
- "광고 도메인의 본질" — 한 박자 쉬고 청중 호기심 유도
- "**첫째 / 둘째 / 셋째**" 각 결정마다 강조 후 한 박자 쉬기
- 각 결정마다 "왜?"를 한 호흡으로 — 결정의 근거가 본문
- 마무리 "각 단계의 결정 근거가 본질에 닿아 있다" 강조 — 이 프로젝트의 메시지

**예상 청중 반응**:
- 현업 데이터 엔지니어 → "Iceberg 채택 근거 = MERGE 멱등성" 한 줄에 즉시 동의. §4에서 더 깊이 풀어달라는 기대.
- 강사 → 강의 데모와 같은 컴포넌트 사용하면서도 **결정의 근거를 자기 언어로 설명**하는 점 평가
- 대표 / 비전공 → "왜 데이터를 두 개 표로 나눴나" 비유로 즉각 이해 (운영자/매니저 분리)
- 동기 → "왜 raw는 Iceberg 아닌가" 한 번에 정리됨

**스피커 노트**:
- 흐름을 그림으로 따라 읽지 말 것 — 그림은 청중이 알아서 본다. **고민 → 결정 → 근거**의 사슬을 말로 풀어내는 게 핵심.
- "Iceberg를 채택한 가장 큰 근거" 표현은 §4 Iceberg 가치 슬라이드의 복선 — 두 슬라이드가 자연스럽게 연결됨
- "두 종류로 분리"는 페르소나(P1/P2) 분리와 매핑 — §1-A에서 깔아둔 "두 사용자가 같은 대시보드의 두 탭"이 여기서 구조 결정으로 이어짐
