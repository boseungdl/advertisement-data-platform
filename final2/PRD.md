# PRD — 광고 캠페인 데이터 플랫폼

> 작성: 2026-05-13 / 갱신: 2026-05-14
> 단계: 워크플로우 1단계 (기획) — v2 (사용자-facing 재정의)
> 양식: 가이드 §2 PRD + 정형 템플릿(규모·NFR·성공기준)

---

## 0. 주제 (한 줄)

광고 캠페인 데이터 플랫폼 — **분석가의 KPI 분석 + 엔지니어의 운영 모니터링을 한 페이지에서** (Iceberg Lakehouse 백엔드).

---

## 1. 문제 정의 (한 줄)

광고 이벤트(노출·클릭·전환)를 정제·집계해 *분석가와 엔지니어가 같이 보는 통합 대시보드*를 제공하고, 100x 성장에도 운영 부담 없이 *늦은 전환까지 자동 반영*하는 데이터 플랫폼.

---

## 2. 페르소나

### P1. 광고 분석가 — "수민" (28세, 광고대행사 분석팀, **메인**)

- **역할**: 캠페인 성과 분석, 매주 임원 보고 자료 작성
- **사용 맥락**: 매일 오전 KPI 확인 + 매주 월요일 보고 자료 준비
- **페인포인트**
  - 매번 캠페인 KPI를 SQL로 새로 짬
  - "어제 캠페인 효과 없네" 판단 후 며칠 뒤 늦은 전환이 들어와 *결과가 뒤집힘*
  - 데이터 신선도가 표에 안 보여서 *얼마나 믿어야 할지* 모름
- **니즈**
  - 캠페인별 KPI(CTR / CVR / 비용 / 전환) 자동 집계
  - 늦게 도착한 전환이 *과거 표에 자동 반영* (어제·오늘 비교 가능)
  - 표 옆에 *신선도 배지* — "마지막 갱신 N분 전"
- **성공 기준**: 캠페인 KPI 보고 준비 시간 **1시간 → 10분**

### P2. 데이터 플랫폼 엔지니어 — "성우" (28세, 데이터플랫폼팀, **메인**)

- **역할**: 광고 데이터 플랫폼 구축·운영, 야간 호출 대응
- **사용 맥락**: 매일 오전 헬스체크 5분 + 야간 알람 시 룬북 참조
- **페인포인트**
  - 작은 파일 누적 → 분석가 쿼리 느려지고 쿼리 비용 폭증
  - 토픽별 신선도 임계가 다른데 알람이 분리 안 됨 (impression 30분 vs conversion 24시간)
  - Compaction이 streaming MERGE와 같은 파티션에서 충돌
- **니즈**
  - 같은 대시보드의 *운영 탭* — 신선도 / 파일 수 / 스냅샷 수 / 매니지먼트 효과 한눈에
  - 매니지먼트(Compaction / Expire / Orphan) 자동화 — 사람이 안 누름
  - 100x까지 깨지지 않는 설계 사고 — 어디부터 깨질지 *문서로* 답변
- **성공 기준**: 매일 헬스체크 **5분 안**, 야간 호출 시 룬북으로 **10분 안** 1차 진단

## 3. 핵심 기능 (MoSCoW)

> **MoSCoW**: Must(필수) / Should(가능하면) / Could(시간 남으면) / Won't(이번 범위 밖)

### Must

- **통합 대시보드** — *분석 탭* + *운영 탭* (같은 페이지)
- 실시간 광고 이벤트 인제스션 (노출 / 클릭 / 전환 *분리 스트림*)
- **메달리온 3계층** — Bronze(raw) / Silver(정제) / Gold(집계)
- **늦은 전환 자동 반영** — 멱등 MERGE로 과거 파티션 갱신
- **Iceberg 매니지먼트 자동화** — Compaction + Expire + Orphan (3종)
- **운영 헬스 쿼리** 5~10개 (Iceberg 메타 기반)
- **캠페인 KPI 집계** — 캠페인×일자 (CTR / CVR / 비용 / 전환)

### Should

- 신선도/품질 임계 알람 (Slack 또는 콘솔)
- 백필 자동화 (룰 변경 시 과거 N일 자동 재계산)
- **100x 스케일 아웃 설계 문서** (구현 X — 평가 기준 ②)

### Could

- Iceberg 스키마 진화 시연
- Criteo 원본 추가 컬럼 활용 (`attribution` / `click_pos` / `cat1~9`)

### Won't (이번 범위 밖)

- **캠페인 운영자에게 데이터 제공** — 기술적으론 가능, 이번 범위 X
- ML 피처스토어
- 실시간 API 서빙
- 멀티 리전 / DR

## 4. 성공 기준 (숫자)

### 기능적

- 일 **100만 이벤트** 처리 (시간당 4만, 초당 12)
- 운영 헬스 쿼리 **7개 이상** 동작
- 통합 대시보드 *분석 + 운영 탭* 모두 동작
- 매니지먼트 자동화 **3종 모두 DAG화**

### 사용자 (페르소나별)

- P1 (분석가): 캠페인 KPI 보고 준비 시간 **1시간 → 10분**
- P1 (분석가): 늦은 전환 자동 반영으로 *결과 뒤집힘* 0회
- P2 (엔지니어): 매일 헬스체크 **5분 이내**
- P2 (엔지니어): 야간 호출 1차 진단 **10분 이내**

### 운영 SLA

- Silver MERGE 후 신선도: impression **30분** / click **60분** / conversion **24시간** 이내

## 5. NFR (비기능 요구사항)

### 처리량 (가이드 §1 기본값)

| 시점 | 일 이벤트 | 시간당 | 초당 |
|---|---|---|---|
| 평소 | 100만 | 4만 | 12 |
| 피크 (2~3배) | — | 12만 | 36 |
| 6개월 10x | 1,000만 | 40만 | 120 |
| 1~2년 100x | **1억** | 400만 | 1,200 |

→ 100x는 *설계 사고만* (평가 기준 ②)

### 저장

- 평균 이벤트 크기 **1KB** → 일 **1GB raw**, 월 30GB, 연 365GB
- Bronze 보존: 30일 hot → 90일 cold → Glacier
- Silver 보존: 1년 / Gold 보존: 무기한

### 가용성·복구

- SLA **99.5%** (월 다운타임 약 3.6시간)
- RTO **30분** / RPO **5분**

### 보안

- PII 없음 (uid 익명 정수)
- IAM 최소 권한, 시크릿은 `.env`(로컬) 또는 AWS Secrets Manager
- 자격 정보 커밋 금지

### 비용

- 월 **5만원 이내** (학습 규모) — 정확 분해는 `ARCHITECTURE.md`

## 용어 풀이

- **Lakehouse**: 데이터레이크(파일 저장소) + 데이터웨어하우스(분석 DB)의 장점을 합친 구조
- **메달리온**: Bronze(원본) / Silver(정제) / Gold(집계) 3계층
- **Iceberg**: 파일 위에서 *테이블처럼* 다룰 수 있게 해주는 오픈 테이블 포맷 (Spark·Athena 모두 읽음)
- **멱등 MERGE**: 같은 작업을 여러 번 해도 결과가 같음 — 재실행 안전
- **늦은 전환(late conversion)**: 광고 전환이 노출보다 *한참 후* 도착 (최대 며칠)
- **Compaction / Expire / Orphan**: Iceberg 운영 3종 — 작은 파일 합치기 / 옛 스냅샷 청소 / 고아 파일 청소
- **MoSCoW**: Must(필수) / Should(가능하면) / Could(시간 남으면) / Won't(이번 범위 밖) 우선순위 분류
